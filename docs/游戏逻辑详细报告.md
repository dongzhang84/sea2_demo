# 大航海 II 游戏逻辑详细报告

这是一份“结果报告”，不是过程日志。

它回答的是三个问题：

1. 这个游戏的逻辑结构到底是什么。
2. 目前已经明确恢复出了哪些内容。
3. 这些内容能具体说明到什么程度。

---

## 1. 总结结论

目前已经可以确定，这个游戏不是简单的线性剧情，而是一个由下列部分组成的状态驱动系统：

- 6 条主人公故事线
- 通用任务系统
- 港口功能系统
- 对话分支系统
- 事件分支系统
- 脚本控制流系统

其中，前四项的“内容层”已经基本可读；最后两项的“机器层”已经看见结构，但还没有全部验明语义。

换句话说：

- 我们已经知道游戏“讲什么”
- 我们已经知道游戏“怎么分支”
- 我们已经开始知道它“怎么跳转”
- 但还没把每个跳转的最终规则全部确认

## 3. 六条主人公线具体是什么样

我们现在能从文本内容里明确看出，6 个主人公不是同一种模板，而是 6 套不同的故事骨架。

### 3.1 贵族出身线

特征：

- 受家族与身份束缚
- 开场就有管家、公爵、公爵夫人等角色
- 主线更偏身世、传说、继承关系

这条线的开场文本已经很明确，属于“被逐出家门后重新起步”的类型。

### 3.2 军官线

特征：

- 和军令、港口调令、职责有关
- 更偏官方身份和任务推进

### 3.3 王室任务线

特征：

- 和国王、宫廷、监督官一类角色有关
- 事件推进更像“被派任务”

### 3.4 探险/制图线

特征：

- 更偏地理、探索和新发现
- 适合作为航海探索的核心体验

### 3.5 底层/还债线

特征：

- 开场更贴近酒馆、欠债、生存压力
- 更像从港口底层开始往上爬

### 3.6 冒险/追梦线

特征：

- 主人公资源较少
- 更强调个人目标和远航动机

### 3.7 这些线说明了什么

说明原作的剧情不是单主线，而是“同一世界规则下的多主人公叙事框架”。

也就是说，游戏的结构不是：

```text
开头 -> 中段 -> 结局
```

而更像：

```text
主人公身份 + 当前位置 + 时间 + 任务状态 + 旗标
-> 选取对应的剧情分支
```

## 4. 通用任务系统已经确认了什么

`Snr0.mes` 不是一条剧情线，而是通用任务系统。

已经明确的任务类型有：

- 运货
- 采购
- 送信
- 剿海盗
- 讨债

这些任务不是单句文本，而是一个完整流程。

### 4.1 典型任务流程

一个通用任务通常包含：

1. 任务发布
2. 接受或拒绝
3. 任务进行中状态
4. 成功完成
5. 超时失败

### 4.2 这说明什么

说明任务不是纯对话，而是“文本 + 条件 + 时间限制 + 结果分支”的组合。

## 7. 事件分支到底已经弄出了什么

这部分是最关键的“结构层”。

### 7.1 已经确认的事实

我们已经把 SNDT 脚本结构整理成：

```text
文件 -> chunk -> 子脚本 -> 派发表 -> 字节码
```

这不是猜测，而是已经可以稳定导出的结构。

### 7.2 已知规模

当前统计是：

- `Snr0-6.dat` 共 **48 个 chunk**
- 共 **186 个 subscript**
- 共 **745 条 dispatch edge**

这些派发表边说明，脚本不是简单顺序播放，而是有入口、有分支、有不同目标块。

### 7.3 静态拓扑下限

目前还能够稳定看到的是：

- 每个 chunk / subscript 的边界
- 每个 subscript 的派发表
- 通过文本引用看到的剧情节点

这已经足够构造第一层拓扑图。

- `fe 07 48`

## 11. 现在还差什么

还没有完全确认的部分主要是：

- 每个 opcode 的准确语义
- 每个条件判断到底在检查什么变量
- 哪些边是跳转，哪些只是参数传递
- 全游戏统一拓扑图
- 运行时如何精确走到某个分支

所以现在的状态是：

```text
内容层基本清楚
结构层很大一部分清楚
执行层还没完全验明
```

## 13. 结论

现在已经能比较准确地说：

> 这个游戏的逻辑不是“一个故事讲完”，而是“多个主人公在同一套世界状态里，通过任务、港口、人物关系和旗标驱动一张分支网络”。

我们现在已经把这张网络的内容层和一部分结构层拿出来了。
