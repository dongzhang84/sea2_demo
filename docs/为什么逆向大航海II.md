# 用 Claude Code 逆向《大航海時代II》—— 过程、方法论与设计洞察

> 这个项目的目标不是重做游戏（做不过原作），也不是建攻略资料库（三十年前就被写透了）。
> 真正要回答两个问题：
>
> 1. **AI 到底怎么帮人做逆向工程？** 现在是 AI 做逆向的窗口期，能不能用一款 1994 年的
>    游戏练手，提炼出方法论？
> 2. **能不能更好地理解老一代设计师的设计智慧？** 那代日式策略游戏设计师，在 640KB
>    内存里塞进了什么样的思考？
>
> 老游戏是练手对象，不是终点。最近更新：2026-05-15。

---

## 一、Claude Code 是怎么逆向这个游戏的

起点是一堆 1994 年的二进制文件——一个 PC-98 时代的 16 位 `Main.exe`、一批
`.lzw` 压缩资源、若干 `.dat` 数据文件——**没有任何文档、没有符号表、没有格式说明**。
全部认知都得从字节里挖出来。整个过程分三个阶段。

### 阶段一：解压资源（破压缩格式）

第一道门是 KOEI 自有的压缩格式（magic `LS11`/`LS10`/`Ls12`）。
- **走过的弯路**：一开始假设它是标准 LZW、LZSS，盲目试了几十种参数组合，全错。
- **转折**：停止盲试，去 GitHub 搜「KOEI 文件格式」，找到社区现成的解码器参考
  （tzengyuxio、JohanLi 等人的研究），比对位流细节——它其实是**位流变长前缀码 +
  每个文件独立的 256 字节字典**。
- 破了压缩，解出：128 张头像、港口 tile、101 张港口地图、世界地图、875 张决斗
  sprite、角色行走帧——所有视觉资源。

### 阶段二：破事件 CG 解码器（核心攻坚）

事件 CG（开场/事件/结局画面）压在一种没人见过的格式里。解码器藏在 `Main.exe` 里。

1. **定位**：用 capstone 写了个反汇编工具，在 298KB 的二进制里找到解码器函数
   （两个变体：写显存的 `0x5e00`、写缓冲区的 `0x61f4`）。
2. **关键决定**：解码器是一段绕人的 nibble 对齐 LZ 压缩。手工把 16 位汇编移植成
   Python 极易出错。**改用 CPU 模拟器（unicorn）直接跑原始机器码**——把代码段
   载入模拟器、按调用约定铺好栈、喂进压缩数据，让真实的 CPU 指令自己跑出结果。
3. **一次跑通**：模拟器输出正确字节，再反推出像素格式（4bpp planar 字节交错）。
4. **大图**：640×400 的整屏 CG 要走另一个写显存的变体——给模拟器补了一个 VGA
   显卡写入模型（latch / 位掩码 / plane 选择）。
5. **自动判别**：发现 CG 有两种编码模式，用一个不变量「VGA 写入数 == 宽×高」
   自动识别该用哪种。
6. **结果**：解出 **231 张事件 CG**（原估计 ~130 张）。

### 阶段三：破文本编码

游戏文本（对话、剧情、UI）存成一种自定义 2 字节编码。
- **线索**：`KoeiCht.txt` 是一张字符映射表。
- **走过的弯路**：一开始把这张表的格式读错了（以为两列都是十六进制文本），
  解出来是「一致的乱码」——同样的输入永远得同样的错字。
- **纠偏**：「结构对、内容错」是**字符集错配**的信号，不是解码错。系统地把映射
  方向、编码（Big5/GBK、大端小端）全扫一遍——最后是「游戏内部码查表 → 取原始
  字节 → 按 GBK 解码」才对。
- **结果**：解出**全部 5325 条游戏文本**——1022 条 UI/对话 + 6 个主人公的完整
  剧情 + 道具文本。整个游戏的剧情现在都能读了。

### 还没破的

事件触发脚本 `Snr*.dat` 是一套自定义字节码虚拟机（`SNDT` 格式）。结构摸清了，
但完整破解要逆向那台虚拟机的解释器——是和 CG 解码器同级、甚至更深的一场攻坚，
仍在进行中。

**用到的工具链**：capstone（反汇编）、unicorn（CPU 模拟）、Python + PIL（解码与
渲染脚本）、Ghidra（辅助分析）。全部脚本在 [`../scripts/`](../scripts)。

## 二、方法论：AI 怎么帮人做逆向

从上面的过程提炼出的原则：

### 1. 能「模拟」就不要「重写」 —— 本次最大的收获

当一段逻辑「复杂但自包含」时，不要逆向它的**算法**，去**模拟执行**它。
你只需要逆向它和外界的**接口**（入口、参数、I/O、依赖的硬件）。
解码逻辑由原始机器码自己执行 —— 结果**天然正确**，根除了「我理解错了」。
这把「完全看懂一个黑盒」降级成了「搭好黑盒的边界」。

### 2. 反汇编是「地图」，不是「逐字翻译」

用反汇编**定位**目标、看清结构（入口/参数/碰了哪些内存/有没有端口 I/O），
而不是把每条指令翻译一遍。看懂结构 → 交给模拟器。

### 3. 先结构，后内容

永远先摸清**容器格式**（偏移表、chunk、子块），再碰内层压缩/编码。

### 4. 先找参考，再动手

盲试参数浪费了大量时间；找社区现成解码器快几十倍。**地毯式检索正是 AI 的强项**。

### 5. 用「冗余」和「不变量」验证

同一份数据用两种方法各解一遍、结果一致 → 可信；找到「写入数 == 宽×高」这类
不变量当判据。

### 6. 「一致但错误」是编码错配的信号

输出「结构对、内容系统性地错」，几乎一定是字符集/映射错配，不是解码逻辑错。
对策是系统地扫所有映射可能性，而不是怀疑算法。

### 7. 估算任务深度，并诚实判断

知道一件事「要挖多深」本身就是方法论。SNDT 虚拟机识别出是深坑后，先摸结构、
如实标注代价，而不是硬钻。

### 8. AI 的强项与陷阱

- **强项**：不知疲倦的交叉检索；几分钟写一个一次性工具；并行试多个假设；
  读得动大段反汇编。
- **陷阱**：会**自信地走错路**（读错 KoeiCht 那次）。所以必须有**验证回路**，
  也需要人在关键处的「这不对」来纠偏。AI 跑得快，但需要校准。

## 三、老一代设计师的设计智慧

从游戏内部结构能读出 1994 年 KOEI 设计师的思路：

### 1. 每个资产自带一本「小码本」

不是用一个全局压缩算法，而是**每个资产携带一份为它定制的小码表**——LS11 压缩
每个文件独立的 256 字节字典、事件 CG 每张图独立的 16 项派发表。在 640KB 里，
自适应的 per-asset 编码能再榨出一截空间。

### 2. 为「事件」造了一台虚拟机

`Snr*.dat` 是字节码脚本——他们专门做了一台小型脚本虚拟机，让设计师把事件/剧本
**当数据来写**，不改引擎、不重新编译。6 个主人公 = 6 套剧本跑在同一套系统上。
「内容与引擎分离」今天是常识，1994 年在 640KB 里这么做是有远见的。

### 3. 游戏的「设计」住在数据表里

101 港 × 12 商品的价格表、30×45 的风/洋流网格、25 种船的属性表——
游戏的平衡与世界**就是这些表**。机制是骨架，数值表是血肉。

### 4. 从不显示整个世界

世界地图是一整张大图，但航海时屏幕只开一个**滚动窗口**看局部——既是内存考虑，
也是注意力设计。

### 5. 压缩是「懂硬件」的

事件 CG 的压缩贴着 VGA 平面显存的结构来设计，把整屏 CG 塞进约 20KB。
不是通用压缩，是为目标硬件量身设计的压缩。

### 6. 简单系统的「涌现深度」

贸易 = 低买高卖 + 区域差价 + 季节波动 + 风影响航程。机制很少，但它们**相乘**——
少数规则交织出足够的策略空间。这是那一代设计的共性：克制的机制，靠交互产生深度。

## 四、还没回答的 / 下一步

- **SNDT 虚拟机**破完，能看到事件触发的**精确条件与分支**——那是「设计智慧」
  最硬核的一层（设计师怎么编排剧情节奏）。
- 方法论目前只来自一款游戏，需要第二个练手样本（如三国志IV）验证可迁移性。
- 「老游戏是练手对象，不是终点」——方法论成熟后，真正的用途是更快地理解
  **任何**遗留系统 / 黑盒二进制。
