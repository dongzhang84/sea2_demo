# 为什么逆向《大航海時代II》—— 方法论与设计洞察

> 这个项目的目标不是重做游戏（做不过原作），也不是建攻略资料库（三十年前就被写透了）。
> 真正要回答两个问题：
>
> 1. **AI 到底怎么帮人做逆向工程？** 现在是 AI 做逆向的窗口期，能不能用一款 1994 年的
>    游戏练手，提炼出方法论？
> 2. **能不能更好地理解老一代设计师的设计智慧？** 那代日式策略游戏设计师，在 640KB
>    内存里塞进了什么样的思考？今天的设计师可以借鉴，但没人系统地挖过。
>
> 老游戏是练手对象，不是终点。本文档记录截至目前从实战中得到的认知。
> 最近更新：2026-05-15。

## 一、AI 辅助逆向工程：方法论

这次破解了三样东西——事件 CG 解码器、文本编码、（部分）事件脚本结构。
从过程里提炼出的原则：

### 1. 能「模拟」就不要「重写」 —— 本次最大的方法论收获

事件 CG 解码器是一段绕人的 nibble 对齐 LZ 压缩，手工把 16 位汇编移植成 Python
极易出错、极慢。换的办法是：**用 CPU 模拟器（unicorn）直接跑原始机器码**。

- 不需要完全理解算法，只需要把**边界**搭好：入口地址、调用约定、输入/输出内存。
- 解码逻辑由原始指令自己执行 —— 结果**天然正确**，不存在「我理解错了」。
- VRAM 解码器同理：只需建一个 VGA 写入模型（write-mode-0 的 latch / 位掩码），
  解码算法本身一行都不用懂。

> **原则**：当一段逻辑「复杂但自包含」时，不要逆向它的算法，去模拟执行它。
> 你只需要逆向它和外界的**接口**。

### 2. 反汇编是「地图」，不是「逐字翻译」

用 capstone + Ghidra 笔记是为了**定位**解码器、看清它的结构（入口、参数、
碰了哪些内存、有没有端口 I/O），不是把每条指令翻译一遍。看懂结构 → 交给模拟器。

### 3. 先结构，后内容

永远先摸清**容器格式**（偏移表、chunk、子块），再碰内层压缩。
Graph/Endgrp/Opgraph 都是「偏移表 + 块」结构——这一层不破，内层无从下手。

### 4. 先找参考，再动手

盲试参数（压缩算法的几十种参数组合）浪费过大量时间。先 GitHub 搜「文件格式」、
找社区现成解码器，比对位流细节快几十倍。**AI 的强项正是这种地毯式检索**。

### 5. 用「冗余」验证

- 同一张图用两个解码器（缓冲区版 + VRAM 版）各解一遍，结果一致 → 模型可信。
- 找**不变量**当判据：「VGA 写入数 == 宽×高」直接用来自动识别两种编码模式。

### 6. 「一致但错误」是编码错配的信号

破文本时一度解出「一致的乱码」（同样的输入永远得同样的错字）。
这种「结构对、内容错」的输出 = 字符集/映射错配，不是解码错。
对策：系统地把所有可能的映射方向、编码（Big5/GBK/大端小端）扫一遍。
最后是「列1 当 GBK 解」而不是 Big5 才对。

### 7. 估算任务深度，诚实判断

SNDT 事件脚本是一套自定义字节码虚拟机。识别出「这是和 CG 解码器同级、甚至更深的
一场攻坚」后，没有硬钻，而是先把结构摸清、如实标注代价。
**知道一件事要多深，本身就是方法论的一部分。**

### 8. AI 的强项与陷阱

- **强项**：不知疲倦的交叉检索；快速写一次性工具（反汇编器、解码器）；
  并行试多个假设；读得动大段反汇编。
- **陷阱**：会**自信地走错路**（KoeiCht 那次）。所以必须有**验证回路**，
  也需要人在关键处的「这不对」信号来纠偏。AI 跑得快，但需要校准。

## 二、老一代设计师的设计智慧

从游戏内部结构能读出 1994 年 KOEI 设计师的思路：

### 1. 每个资产自带一本「小码本」

不是用一个全局压缩算法，而是**每个资产携带一份为它定制的小码表**：
- LS11 压缩——**每个文件**独立的 256 字节字典。
- 事件 CG——**每张图**独立的 16 项派发表。

在 640KB 里，自适应的 per-asset 编码能再榨出一截空间。这是「为每块数据量身定制」
的思路，而不是「一把锤子敲所有钉子」。

### 2. 为「事件」造了一台虚拟机

`Snr*.dat` 是 `SNDT` 格式的**字节码脚本**——他们专门做了一台小型脚本虚拟机，
让设计师把事件/剧本**当数据来写**，不用改引擎、不用重新编译。
6 个主人公 = 6 套剧本脚本跑在同一套系统上。
「内容与引擎分离」今天是常识，1994 年在 640KB 里这么做是有远见的。

### 3. 游戏的「设计」住在数据表里

101 港 × 12 商品的价格表、30×45 的风/洋流网格、25 种船的属性表——
游戏的平衡与世界**就是这些表**。设计师本质上在调一张大电子表格。
机制是骨架，数值表是血肉。

### 4. 从不显示整个世界

世界地图是一整张大图，但航海时屏幕只开一个**滚动窗口**看局部。
这既是内存考虑，也是**注意力设计**——让玩家聚焦在「此刻所在的海域」。

### 5. 压缩是「懂硬件」的

事件 CG = 4bpp planar + nibble 粒度的 LZ + 0x38 转义。
压缩方案贴着 VGA 的平面显存结构来设计——把 640×400 的整屏 CG 塞进约 20KB。
不是通用压缩，是**为目标硬件量身设计**的压缩。

### 6. 本地化做了一个「间接层」

繁体中文版用 `KoeiCht.txt` 做字符映射——游戏文本存内部码，经一张表换成字库字模。
这是一层**本地化架构**：换字符集不用动引擎。

### 7. 简单系统的「涌现深度」

贸易玩法 = 低买高卖 + 区域差价 + 季节波动 + 风影响航程。
机制很少，但它们**相乘**——少数规则交织出足够的策略空间。
这是那一代设计的共性：克制的机制数量，靠交互产生深度。

## 三、还没回答的 / 下一步

- **SNDT 虚拟机**没破完——破了它，能看到事件触发的**精确条件与分支**，
  那是「设计智慧」最硬核的一层（设计师怎么编排剧情节奏）。
- 方法论还需要更多样本——目前的结论来自一款游戏，下一个练手对象（如三国志IV
  的 `MAINMAP.S4`）能验证哪些原则可迁移。
- 「老游戏是练手对象，不是终点」——方法论成熟后，真正的用途是更快地理解
  **任何**遗留系统 / 黑盒二进制。
