# 大航海时代 II 逆向流程

这份文档写的是“我们是怎么做的”，不是“做出了什么”。

它把整个逆向工作拆成可重复的步骤。后面换别的老游戏时，也可以直接照这个流程走。

## 0. 先定目标

先回答两个问题：

- 我们是要复刻原作，还是提炼方法和设计智慧？
- 这次逆向最后要产出什么？

对《大航海时代 II》来说，答案已经定死：

- 不是复刻原作。
- 不是做资料库。
- 是提炼 AI 逆向工程的方法论，以及老一代设计师的结构化设计智慧。

## 1. 做文件盘点

先把所有原始文件扫出来，搞清楚：

- 哪些是压缩资源。
- 哪些是数据表。
- 哪些是文本。
- 哪些是可直接读的脚本或容器。
- 哪些根本不是当前优先级。

对这个项目来说，第一批重点是：

- `*.lzw`
- `Message.dat`
- `Snr*.dat` / `Snr*.mes`
- `Main.exe`
- `KOUKAI2.DAT`
- `Windcur.dat` / `Za_dat.dat` / `Transit.dat`

## 2. 先把能看见的东西抽出来

先不谈解释器，先让内容“能看”。

这里做的是：

- 解压资源。
- 渲染头像、港口图、世界图、船图、角色图。
- 把文本原样抽成可读文件。
- 把数据表先变成 JSON 或 Markdown。

这一步的作用是把“二进制噪声”变成“可审阅材料”。

## 3. 建第一层索引

把原始材料按最稳定的维度排起来：

- 资源索引
- 文本索引
- 数据表索引
- 人物索引
- 地点索引

这一步不急着下结论，只是先把东西放到可以查的位置上。

## 4. 识别结构边界

开始找“块”和“层”。

对剧情容器来说，重点是：

- 文件
- chunk
- subscript
- dispatch table
- opcode

对《大航海时代 II》来说，这一步的关键产物是 SNDT 拓扑：

- 先知道一条脚本怎么切块。
- 再知道一个子脚本怎么派发。
- 再知道哪些文本是可信引用。

## 5. 做静态拓扑

如果动态解释器还没破，先做静态下限。

原则是：

- 宁愿标注不完整，也不要装成全懂了。
- 对每条结构边做可信度分级。
- 先把“已经确定的骨架”画出来。

这一步的产物是拓扑图、索引表、文本引用表。

## 6. 做动态验证

静态结构只能说明“像什么”，不能说明“真的是什么”。

所以要再做一层验证：

- 断点。
- 寄存器观察。
- 调用栈。
- 内存快照。
- 最小 patch。

这里的目标是把猜测变成证据。

## 7. 把事件还原成状态机

剧情不是句子堆，而是状态流转。

所以要继续把它压成：

- 谁触发了什么。
- 在哪里触发。
- 前提是什么。
- 状态怎么变。
- 下一跳是什么。

这一步会把文本、人物、地点、任务收束成一张图。

## 8. 抽出设计原语

当拓扑稳定以后，不再只看“剧情是什么”，而是看“设计是怎么运转的”。

对这类老游戏，最有价值的设计原语通常是：

- 身份驱动
- 地点驱动
- 资源驱动
- 关系驱动
- 事件回流
- 跨线交叉
- 有限信息下的世界感

## 9. 压成设计稿和 MVP

逆向不是终点。

最后要把这些结构压成可实现的规格：

- 设计稿
- MVP 规格
- 原型实现

这样逆向出来的东西才会变成新项目的输入，而不是只停在研究文档。

## 10. 再压成方法账、设计账和样板

当逆向结果稳定以后，再把成果分成三份：

- **方法账**：记录我们怎么拆、怎么索引、怎么验证。
- **设计账**：记录我们从老游戏里提炼出的设计原语。
- **样板切片**：选一段代表性剧情，把从原文到拓扑再到设计结论的全流程写成模板。

## 11. 这次项目的实际顺序

《大航海时代 II》这次实际走过的顺序是：

1. 先解资源和图像。
2. 再抽数据表和文本。
3. 再建 SNDT / 剧情拓扑。
4. 再做动态验证。
5. 再把结果汇总成剧情说明、设计稿和 MVP 规格。
6. 最后把这些材料收束到 `README.md`、`ROADMAP.md` 和主文档里。

## 12. 一句话总结

**先盘点，后抽取；先索引，后建图；先静态，后动态；先还原结构，再提炼设计。**
