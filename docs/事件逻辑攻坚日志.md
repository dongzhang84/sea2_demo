## 初步发现（2026-05-15）

### KoeiCht.txt = 文本编码的钥匙
每行格式：`<2字节码> TAB <十进制索引> CRLF`
- 例：`a1a1 \t 8130`，`a3ac \t 8131`，`a1a2 \t 8132`
- 2 字节码是 **Big5** 编码；十进制索引顺序递增（8130, 8131, ...）
- 即：**字库字模序号 ↔ Big5 字符** 的对照表。
- 游戏文本存字模序号，查此表 → Big5 → Unicode 即可读出中文。

### 文件结构
- `Message.dat`：开头 u16 偏移表（首偏移 0x7fc → 1022 条），其后文本。
- `Snr*.mes`：开头 u32 BE 偏移表，其后文本。
- `Snr*.dat`：`SNDT` magic + 头 + u32 BE 偏移表 → 脚本块。
- `Event*.dat`：实为 CG 图（含 0x38 escape、0x0GRB 值），非脚本——归到 CG 一类。

## 进度

### 2026-05-15 — 文本编码破解（核心突破）

走了点弯路才认清 `KoeiCht.txt` 的真实格式：
- 每行 = `<列1: 原始 2 字节>` + TAB + `<列2: ASCII 十六进制串>` + CRLF。
- 列1 是 **GBK 编码**的字节；列2 是游戏的内部 2 字节码（十六进制）。

**解码配方**（`scripts/decode_text.py`）：
1. 游戏文本里读 2 字节**大端**码。
2. 在 `KoeiCht.txt` 查列2 == 该码 → 取列1 原始字节。
3. 列1 字节按 **GBK** 解码 → 繁体中文。
4. ASCII 字节（含 `%s` `$n` `$d20` 等占位符）原样保留。

**成果**：
- `Message.dat` 1022 条 UI/对话全部读出（"歡迎光臨！" "本港不允許投資．" …）。
- `Snr0.mes` 192 条 = 通用任务对话（送货/剿海盗/讨债…）。
- `Snr1-6.mes` = **6 个主人公的剧情对话**（Snr1 共 1176 条：管家麥克、
  公爵夫人、普萊斯特·約翰國… João 线剧情）。
- 占位符：`$n`=人名 `$s`=姓 `$r32`=地名/物名引用 `$d20`=数值引用。

### 2026-05-15 — SNDT 脚本结构分析

`Snr*.dat` 文件结构（已确认）：
```
0x00  magic "SNDT"
0x04  头 (0x80 00 ... 共 12 字节)
0x10  u32 BE 偏移表 → 各 chunk, 以 0xffffffff 结尾
      Snr0 有 13 个 chunk
chunk 内部:
  开头若干 u16 = 子脚本入口偏移 (chunk 内), 以 ffff 结尾
    (Snr0 chunk0 有 6 个子脚本)
  其后是字节码
```

字节码是**一套自定义虚拟机的指令流**：
- 1 字节 opcode + 不定长操作数。
- 见到的 opcode：`a3`(后跟 byte+u16)、`dc` `8c`(4字节记录) `4c` `eb`
  `0c` `09` `0a` `10` `8f` `1d` …
- `0xffffffff` 作分隔/结束。

**全量解码需要逆向那个解释器**——它在 `Main.exe` 里，无符号
（`DKK2E.exe` 经查只是 .NET 做的翻译工具，不含 VM）。
Main.exe 里有 29 处跳转表派发（`jmp cs:[bx+d16]`），SNDT 解释器是其一。

完整破解 = 找到派发点 → 逆向 30-50 个 opcode handler → 写反编译器。
这是和 CG 解码器同级的一场攻坚。

### 2026-05-15 — SNDT VM 解释器排查（结果：静态法未定位）

系统排查了一轮：
- 字节码频率分析：高频字节 `0x8c 0xc8 0xd0 0xc0 0xdc` → 确认是 VM，
  opcode 很可能落在高位段（推测 0xc0–0xff）。
- 逐个查了 Main.exe 全部 **29 处跳转表派发**的 case 数与派发来源。
- 最像的候选 `0x382c1`：在 `[0xb87a]-0xc0` 上派发 64 路（正好覆盖 0xc0–0xff）。
  但反汇编它的跳转表项 → 有的指向函数中段，**不是干净的 handler 入口**，
  判定它不是 SNDT VM。
- 其余 28 处 case 数偏小（5–30）、来源是固定全局/小枚举，不符合脚本 VM。

**结论**：SNDT 解释器**没有用可识别的跳转表派发**（可能是比较链 / 多张表 /
结构与预期不同）。纯静态反汇编这一轮没能定位它。

**诚实判断**：SNDT VM 是研究级深坑。可靠的破解只剩**动态法**——emulate
Main.exe、追踪运行时哪段代码读 SNDT 字节。但这要先搭 DOS 运行环境
（INT 21h、PSP、把游戏跑到事件触发），是一摊多轮工程，且不保证短期拿下。

结构层（文件→chunk→子脚本→字节码）已完全摸清。文本层（5325 条）已全解。
事件「逻辑」的人类可读层（剧情/任务/对话）其实已经拿到——SNDT 字节码是其上
机器精确的触发条件层。
